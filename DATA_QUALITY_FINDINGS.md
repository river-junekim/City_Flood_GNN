# 데이터 품질 진단 보고서 — 도심 침수 예측 프로젝트

> 작성일: 2026-06-17
> 대상: 서울시 하수관로 수위 센서 + 도로노면 수위 센서 데이터
> 결론 요약: **전처리·변환은 정상. 그러나 도로 침수 라벨(`flood_flag`)의 약 95.7%가 센서 baseline/노이즈로 인한 가짜이며, 진짜 침수 신호는 매우 희소(17,473건/22센서/불균형 1:1297). 원래 프로젝트 전제("하수관로로 도로 침수를 180분 전 예측")는 현 데이터로 뒷받침되지 않음.**

---

## 0. 한눈에 보기

| 항목 | 결과 |
|------|------|
| CSV/TXT → parquet 변환 | ✅ **정상** (원본 = parquet 정확 일치, 2회 검증) |
| `flood_flag = level>0` 라벨 | ❌ **약 95.7%가 가짜** (센서 baseline/stuck) |
| 진짜 침수 (정제 후) | **17,473건 / 22센서 / 불균형 1:1297** |
| 하수 → 도로 선행성 | ❌ **없음** (하수는 후행 지표, 공간 GNN 이득 null) |
| 센서 노이즈 | ❌ 연중 ~11% 비영점 (계절 신호 미약) |
| 통신상태 필터 | 효과 없음 (전부 통신양호) |
| 관악구(고객 타겟) 단독 학습 | ❌ 이벤트 희소 → 서울 학습 필요 |

---

## 1. 변환·처리는 정상이다 (먼저 배제)

데이터가 이상해 보여 변환 단계를 의심했으나, **원본 CSV와 parquet을 직접 대조한 결과 완전히 일치**했다.

- **2024-01 하수**: 원본 `측정수위` 분포 = parquet `level` 분포 (mean 0.15, max 3.58, 센서 22-0014 첫 5개 모두 0.05)
- **2024-07 하수**: 원본 최댓값 행 = `07-0009, 2024-07-04 15:48, 11.2, 통신양호` = parquet 동일

→ **변환 버그 아님.** 데이터의 이상한 값들은 **센서가 실제로 그렇게 기록한 값**이다. 재처리로 해결되지 않는다.

> 참고: 하수 `측정수위`는 **미터(m)**, 도로 `level`은 **센티미터(cm)** 단위. 두 도메인 값을 같은 숫자로 비교하면 안 됨.

---

## 2. 핵심 문제 — 침수 라벨의 95.7%가 가짜

### 2-1. 증상: 물리적으로 불가능한 분포
`flood_flag = (level > 0)`로 만든 도로 침수 라벨 총 **2,149,356건**의 월별 분포:

```
1월 295,562  ≈  7월 322,566      ← 한겨울이 장마 절정과 맞먹음(불가능)
겨울(12·1·2) = 570,409 = 전체의 26.5%
```
실제 강우 침수는 여름(6~9월)에 70~90% 집중돼야 하나, 여름은 38%뿐.

### 2-2. 원인: 센서 baseline + 과민한 임계
- 도로 센서는 **마른 상태에서도 0이 아닌 baseline**(예: 개봉동 403-59는 한겨울 1월에 값 "1"을 56% 기록)을 출력한다.
- 원본 값이 **이산적·저수준**(개봉동: 0·1·6·11)이고, 센서마다 작은 값(0.5·1·6) 또는 큰 값(봉천동 11·26, 화곡동 31)에 **stuck**돼 있다.
- `flood_flag = level>0` 이 이 **baseline/노이즈를 전부 침수로 카운트**했다.

### 2-3. 정량: 오염은 47개 센서에 집중
센서별 "최빈 3개 값 점유율 ≥ 80%"를 stuck 지표로 본 결과:

| 구분 | 센서 수 | 침수 건수 | 비중 |
|------|--------|----------|------|
| **stuck 센서** | 47 | 2,057,449 | **95.7%** |
| 정상 센서 | 37 | 91,907 | 4.3% |

→ 가짜 침수의 거의 전부가 **47개 stuck 센서**에서 나온다.

### 2-4. GIS 제원으로 교차 확인
도시침수계(노면) 제원: 측정 가능 범위 0~96cm인데, 오염 센서들의 **기왕최대(역대 최고 수위)는 1~26cm로 매우 낮음**. 즉 이 센서들은 애초에 큰 침수를 겪은 적이 없으며, 낮은 baseline이 라벨을 오염시켰다.

### 2-5. 통신상태로는 못 거른다
오염 센서를 포함해 데이터가 **전부 `통신양호`**로 기록된다. 통신양호 센서도 stuck/baseline 값을 내보낸다(예: 07-0009은 통신양호인데 11.2m stuck). → 통신상태 필터는 무효.

---

## 3. 진짜 침수 신호의 실제 규모

`level>0` 폐기 후 **센서별 baseline + 강우 검증**으로 진짜 침수를 정의:
> 진짜 침수 = `level ≥ (센서별 baseline + 5cm)` **AND** `직전 6시간 누적강우 ≥ 1mm`

| 단계 | 건수 |
|------|------|
| 기존 flood_flag(level>0) | 2,149,356 |
| 후보(baseline+5cm 이상) | 120,835 |
| **진짜 침수(+강우 동반)** | **17,473** |

- **진짜 침수 보유 센서: 22 / 100개**
- **진짜 불균형: 1:1297** (기존 가정 1:20과 차원이 다름)
- 단, 정제 후에도 월별이 완전히 깨끗하진 않음(7월 최고지만 겨울·봄 잔존) → 일부 오염 잔류 가능

**강우 검증 근거**: 강우 관측소 48개, 도로→최근접 관측소 중앙값 1.45km. stuck 센서 제외 시 정상 센서의 여름 침수 강우동반율이 22% → 53.5%로 급상승(물리적으로 타당해짐).

---

## 4. 하수관로는 도로 침수의 선행 지표가 아니다

교차상관 분석(±180분, 연속축 shift 후 이벤트 평가):
- raw 수위: 이벤트 최빈 lag **-1**(하수가 도로보다 10분 **후행**), 60분 초과 선행 0건
- 변화율(level_diff): 강한 상관 중 **선행 0건**, 최빈 lag -2/-3(더 후행)
- 하수 월별 분포가 **계절 무관**(7월 ≈ 1월) — 폭우 반응이 분포 body에 거의 안 나타남

→ **하수는 후행 지표.** 도로 표면수가 먼저 차고 하수가 뒤따라 반응한다. 따라서 **sewer→road 공간 GNN은 예측 lead를 더하지 못한다**(공간 이득 null). 이는 별도 실험에서 Baseline-LSTM(공간 정보 없음)이 GNN과 동등했던 것과도 일치.

> ⚠️ 단, 위 교차상관도 오염된 라벨/이벤트 위에서 계산된 것이므로, 깨끗한 라벨로 재검토 시 일부 달라질 수 있음. 그러나 하수 분포의 계절 무반응성은 라벨과 무관하게 관측됨.

---

## 5. 관악구(고객 타겟) 범위 이슈

고객 요구는 **관악 지역 한정** 모델(단위배수구역 기준). GIS 매칭 결과 관악 타겟은 **도로 6개 + 하수 17개** 수준.
- 관악 도로의 학습기간(2024) 진짜 침수 이벤트가 매우 적고, 핵심 센서(봉천동 911-14)는 stuck(미수신).
- → **관악 단독 학습 불가.** 서울 전체로 학습 후 관악을 추출·평가하는 전이 방식이 필요.

## 7. 결론

이 진단의 핵심 가치는 **95% 가짜 라벨 위에 모델을 올리는 것을 막은 것**이다. 변환·코드는 깨끗하며, 문제는 **센서 데이터에 내재한 noise/baseline + 잘못된 침수 정의(`level>0`)** 다.

진짜 침수 신호는 작지만(17.5k/22센서) 존재한다. 따라서 프로젝트는 **폐기가 아니라 재정의**가 필요하다 — 강우를 핵심 동인으로, 깨끗한 라벨 위에서, 현실적 목표(단기 탐지)로. 그 전에 **고객과의 기대치 재조정**이 선행되어야 한다.
